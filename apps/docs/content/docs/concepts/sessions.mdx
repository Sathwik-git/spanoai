---
title: Sessions & Audit
description: Scope context and messages to a run, manage membership and cancellation, and replay an immutable trail.
---

A **session** is the unit of isolation — a single run, conversation, or job. Context
entries, messages, and artifacts all live inside a session, and agents in different
sessions never see each other's state.

## Lifecycle

```ts
await client.sessions.create({ sessionId: "run-1", ttlSeconds: 3600, metadata: { task: "diligence" } });
const s = await client.sessions.get("run-1");
const all = await client.sessions.list();
```

Sessions can carry a TTL and arbitrary metadata. A session is **automatically
registered** the first time any agent writes context or dispatches a message into it —
so you don't have to create one up front, though doing so lets you set TTL/metadata
explicitly.

<Callout type="info">
  Auto-registration is idempotent: an explicit `create` is never clobbered by the
  implicit touch on first write.
</Callout>

## Membership

Agents can join and leave a session, and you can list members. Membership is useful for
broadcast targeting and for dashboards that show "who's in this run".

```ts
await client.sessions.join("run-1", "researcher");
await client.sessions.leave("run-1", "researcher");
```

## Cancellation (abort)

A session carries a cooperative **abort flag**. Setting it signals every agent in the
run to stop — long-running agents check the flag and bail out cleanly, rather than
being killed mid-write.

## Audit log

Every meaningful action — context writes, message dispatches, session start/stop — is
appended to a durable **audit log** in Postgres. It is the source of truth for "who did
what, when", and it is queryable:

```ts
const trail = await client.audit.list("run-1");
```

### Resilience

Audit writes never block the hot path. If Postgres is briefly unavailable, entries are
buffered to a Redis list and drained by a background job once the database returns — so
an audit outage degrades durability, not availability.

### Replay

Because the log is ordered and complete, a run can be **replayed** — reconstructing the
sequence of context states and messages. This is the basis for debugging coordination
bugs and for time-travel views in the dashboard.

<Callout type="warn">
  The audit log is append-only by design. Treat it as immutable history, not mutable
  state.
</Callout>
